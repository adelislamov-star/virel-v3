## 1. Что такое Quick Upload

Quick Upload — инструмент для операторов Virel для быстрого создания профиля модели
из анкеты и фотографий за один экран без ручного ввода данных.

Поток работы оператора:

1. Оператор перетаскивает фото и анкету (PDF, DOCX, TXT, XLSX) в drop zone
2. Система параллельно запускает два AI процесса:
   - parse-anketa — Claude читает анкету и извлекает все данные
   - sort-photos — Claude сортирует фото по качеству и назначает роли
3. Оператор видит Preview: предупреждения, распознанные данные, вводит имя
4. Нажимает Create — система создаёт профиль в БД, загружает фото в Cloudflare R2
5. Done — ссылка на готовый профиль

Технический стек:
- Frontend: Next.js 14, React, TypeScript
- Backend: Node.js serverless на Vercel (maxDuration 120s)
- AI: Claude Opus 4.5 через Anthropic API
- БД: PostgreSQL на Neon через Prisma ORM
- Хранилище фото: Cloudflare R2 (WebP конвертация, watermark, thumbnail)
- Деплой: Vercel с автодеплоем из GitHub main

Ключевые файлы:
- `src/app/admin/quick-upload/page.tsx` — весь фронтенд, state machine
- `src/app/api/v1/models/quick-upload/route.ts` — основной backend (~1200 строк, 8 шагов)
- `src/app/api/admin/parse-anketa/route.ts` — AI парсинг анкеты
- `src/app/api/admin/sort-photos/route.ts` — AI сортировка фото
- `src/lib/r2.ts` — загрузка в Cloudflare R2, watermark, thumbnail
- `src/lib/publish-validation.ts` — readiness check профиля
- `prisma/schema.prisma` — схема БД

## 2. Архитектура backend pipeline

Quick Upload route выполняет 8 последовательных шагов:

Step 1 — создание модели (prisma.$transaction начало)
Step 2 — привязка сервисов (tx.modelService.create)
Step 3 — создание ставок (tx.modelRate.createMany + калькулятор)
Step 4 — адрес (tx.modelAddress.upsert)
Step 5 — преференции (tx.model.update)
Step 5.5 — определение локации (fuzzy поиск TransportHub → District → Location)
--- транзакция заканчивается здесь ---
Step 6 — загрузка фото в R2 (вне транзакции, external I/O)
Step 7 — сохранение примера в AIParseExample для обучения
Step 8 — QuickUploadLog + auditModelReadiness

Steps 1–5.5 атомарны: если что-то падает — откатывается всё, профиль не создаётся.
Step 6 (фото) вне транзакции: при полном провале — профиль удаляется. При частичном —
профиль остаётся, оператор видит предупреждение о неполной загрузке.

## 3. Как работает AI парсинг

### 3.1 parse-anketa

Когда оператор загружает анкету, система не просто отправляет текст в Claude.
Перед запросом происходит следующее:

Шаг 1 — загрузка обучающих примеров.
Из таблицы `AIParseExample` загружаются топ-5 реальных анкет которые уже парсились.
Сортировка: сначала самые исправляемые операторами (correctionCount desc),
потом с низким qualityScore (там видно где AI ошибался), потом новые.

Шаг 2 — few-shot обучение.
Примеры вставляются в prompt в формате "вот анкета — вот правильный JSON ответ".
Claude видит реальные примеры до того как начнёт парсить новую анкету.

Шаг 3 — блок известных ошибок.
Если у загруженных примеров есть operatorCorrections, система собирает уникальные
паттерны ошибок и добавляет блок в конец system prompt:

  KNOWN PAST AI MISTAKES — avoid repeating these:
  - Field "bustType": AI returned "null" but correct value was "silicone"
  - Field "disabledClients": AI returned "true" but correct value was "false"

Шаг 4 — парсинг.
Claude получает: инструкции + примеры + список своих прошлых ошибок + новую анкету.
Возвращает JSON с полями профиля.

Шаг 5 — сохранение нового примера.
После успешного Create текст анкеты и итоговый JSON сохраняются в AIParseExample.
Следующий парсинг уже учитывает этот случай.

### 3.2 sort-photos

Параллельно с parse-anketa запускается sort-photos.
Claude получает base64 превью фотографий и возвращает порядок сортировки с ролями:
primary (главное фото), gallery (галерея), other.
Результат используется для отображения Preview и для порядка загрузки в R2.

### 3.3 Что парсится из анкеты

Основные поля: name, age, height, weight, nationality, languages, orientation,
smokingStatus, tattooStatus, piercingTypes, eyeColor, hairColor, hairLength,
bustSize, bustType, dressSize, feetSize.

Ставки: все временные слоты (30min, 45min, 1h, 90min, 2h, 3h, overnight, extra-hour)
с разделением incall/outcall.

Сервисы: массив объектов { code, enabled, extraPrice } по списку из ~60 сервисов.

Преференции: workWithCouples, workWithWomen, blackClients, disabledClients,
аэропорты (Heathrow, Gatwick, Stansted).

Адрес: street, city, postcode.
Локация: название станции метро для определения района.

## 4. Feedback loop — как система учится на ошибках операторов

### 4.1 Проблема которую решает feedback loop

До внедрения: AI учился только на своих же примерах. Если AI ошибся с bustType —
он продолжал ошибаться. Оператор исправлял вручную, но система ничего не запоминала.
Одни и те же ошибки повторялись на каждой новой анкете.

### 4.2 Как работает петля обратной связи

Когда оператор открывает профиль после Quick Upload и вручную исправляет поля,
а потом нажимает Save — срабатывает PATCH /api/v1/models/[id].

В этот момент:

1. Система находит aiParseExampleId профиля — ссылку на пример из Quick Upload.

2. Берёт 17 tracked fields и сравнивает что AI распознал (parsedOutput)
   с тем что оператор поставил (тело запроса).

3. Все расхождения записываются в operatorCorrections:
   { "bustType": { "ai": null, "operator": "silicone" } }

4. correctionCount инкрементируется — чем больше раз пример исправляли,
   тем он ценнее для обучения следующих парсингов.

5. qualityScore пересчитывается: 1.0 = AI угадал все поля правильно,
   0.0 = AI ошибся везде. Формула: 1 - (количество ошибок / количество заполненных полей).

Tracked fields (17 штук):
bustType, bustSize, smokingStatus, tattooStatus, disabledClients, blackClients,
workWithCouples, workWithWomen, orientation, nationality, age, height, weight,
dressSize, feetSize, hairColor, eyeColor.

### 4.3 Как это влияет на следующие парсинги

При следующем Quick Upload few-shot примеры берутся именно те, которые:
- Чаще всего исправлялись операторами (correctionCount desc) — самые информативные
- Имеют низкий qualityScore — показывают типичные ошибки AI
- Новые — свежий контекст

В system prompt добавляется блок KNOWN PAST AI MISTAKES со всеми уникальными
паттернами ошибок из всех corrections всех примеров (до 20 паттернов).

### 4.4 Нарастающий эффект

5 анкет — AI уже знает про bustType и disabledClients.
20 анкет — AI стабильно парсит все проблемные поля правильно.
50 анкет — система работает почти без ручных правок операторов.

### 4.5 Текущие ограничения feedback loop

Не учится на исправлениях ставок (rates не в tracked fields).
Не учится на исправлениях сервисов (isExtra и т.д.).
Нет дашборда где видно как растёт qualityScore со временем.
Это следующий этап развития системы.

### Fix 6 — Race condition дубль-чека [P0]
Коммит: e9336f0 | Верифицировано: в diff на GitHub

Два оператора одновременно проходили проверку дублей (read без lock) и создавали

Решение: prisma.model.create обёрнут в try/catch. При ошибке P2002 (unique constraint
на slug) возвращается 409 DUPLICATE_SLUG вместо 500. Убран дублирующий @@index([slug]) —
@unique уже был на поле и создаёт индекс автоматически.
Файлы: prisma/schema.prisma, src/app/api/v1/models/quick-upload/route.ts


### Fix 7 — Fuzzy location matching [P2]
Коммит: a665ca1 | Верифицировано: в diff на GitHub

Поиск транспортного хаба шёл по точному совпадению имени. "Baker St" не находил
"Baker Street", "King's Cross" не находил "Kings Cross". primaryLocationId не
записывался для большинства анкет — оператор ставил локацию вручную каждый раз.

Решение: функция normalizeStation() убирает точки и апострофы, разворачивает
"St" в "Street", приводит к lowercase. Двухуровневый поиск: сначала exact match
по нормализованному slug и name (case-insensitive), потом contains-fallback.
Файл: src/app/api/v1/models/quick-upload/route.ts


### Fix 8 — Защита от reload страницы во время uploading [P1]
Коммит: 8dd4396 | Верифицировано: полностью на живом сайте

Reload страницы во время Create — запрос продолжает выполняться на сервере.

### Fix 9 — Транзакция Steps 1–5.5 + partial upload warning [P0]
Коммит: 44458cc | Верифицировано: CI green, в chunk на живом сайте

Главный баг системы. Steps 1–8 были независимыми await без транзакции.
Если R2 падал на 4-м фото из 8 — профиль существовал в БД с частичными данными.

### Fix 14 — Баги из реальной анкеты Marsalina [P1]
Коммит: 8867932 | Верифицировано: CI green

Обнаружены при первом реальном тесте системы на боевой анкете. До этого момента
система тестировалась только синтетически.

isExtra не проставлялся.
extraPrice записывался в БД при создании ModelService, но isExtra всегда оставался
false. Галочка "Extra service" в UI профиля была всегда пустой.
Решение: isExtra: svcExtraPrice != null && svcExtraPrice > 0

Bust Type не парсился.
AI получал строку "silicone 34C" и не разделял тип от размера.
breastSize приходил правильно, breastType оставался пустым.
Решение: разделены инструкции в AI prompt. Добавлен явный пример:
"silicone 34C" → breastType: "silicone", breastSize: "34C"

disabledClients defaulted to true.
AI игнорировал явное "no" в анкете и выставлял disabledClients: true по умолчанию.
Решение: добавлено явное уточнение в prompt:
"IMPORTANT: disabledClients means does the model accept disabled clients.
If answer is no → false. Do NOT default to true."

Многочасовые ставки не вычислялись.
3h ставки были пустыми если не написаны в анкете явно.
Overnight outcall не копировался из incall.
Формула которая используется в агентстве:
2h = 1h incall + extra hour
3h = 1h incall + extra hour × 2
(аналогично для outcall)
Overnight: одна цена для incall и outcall.

Решение: калькулятор после mapRatesToRows:
- Если есть 1h incall и extra hour, но нет 2h incall — вычисляется автоматически
- Если есть 1h outcall и extra hour, но нет 2h outcall — вычисляется автоматически
- То же самое для 3h
- Если есть overnight incall но нет overnight outcall — копируется
