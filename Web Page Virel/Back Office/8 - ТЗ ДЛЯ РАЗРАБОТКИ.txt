# 2. Общие технические правила

## 2.1 Стек

Использовать текущий стек:

* Next.js 14
* TypeScript
* PostgreSQL
* Prisma
* Tailwind
* Stripe
* Vercel

## 2.2 Общие ограничения

* не ломать текущие рабочие flows
* все миграции только аддитивные
* все новые поля с default или nullable
* никакого hard delete для ключевых бизнес-сущностей
* все денежные значения хранить как Decimal / Numeric
* все критичные действия писать в AuditLog
* все длительные операции выполнять через JobQueue
* все state-changing endpoints валидировать и логировать
* все ручные overrides только с reason_code
* все list endpoints с pagination
* все новые admin страницы в текущем dark UI pattern

## 2.3 Naming

* в БД использовать snake_case
* в Prisma / TypeScript использовать camelCase mapping
* naming должен быть единообразным по всему проекту

## 2.4 Денежные типы

Все поля ниже хранить как Decimal / Numeric:

* total_spent
* amount
* value_amount
* total_amount
* commission_amount
* payout_amount
* refunded_amount
* compensation_amount
* price_monthly
* revenue_amount
* avg_price
* margin_impact
* cancellation_fee

## 2.5 Soft delete

Обязательно добавить soft delete поля:

* deleted_at
* deleted_by

В таблицы:

* clients
* bookings
* models
* reviews
* incidents
* membership_plans
* pricing_rules
* notification_templates

## 2.6 Audit

Все критичные действия обязаны создавать запись в AuditLog.

Минимальный список:

* client.merge
* client.identity.added
* client.risk_status.changed
* booking.status.changed
* booking.disputed
* payment.refunded
* payment.disputed

* availability.block.removed

* operations_feed.assigned
* operations_feed.resolved
* incident.created
* incident.status.changed
* review.status.changed
* review.visibility.changed
* membership.subscription.created
* membership.subscription.cancelled
* membership.benefit.applied
* pricing.rule.created
* pricing.rule.updated
* pricing.rule.disabled
* notification.template.updated
* job.retried
* retention.action.created
* retention.action.completed

## 2.7 API response format

Успех:

```json
{ "data": { } }
```

Ошибка:

```json
{ "error": { "code": "string", "message": "string" } }
```

## 2.8 Общие требования ко всем новым API

* RBAC check
* input validation
* pagination для list endpoints
* sorting support там, где нужен список
* filtering support там, где нужен список
* standardized error codes
* audit logging для critical actions
* idempotency для повторяемых state-changing операций
* permissions-aware responses
* reason_code для manual critical actions
