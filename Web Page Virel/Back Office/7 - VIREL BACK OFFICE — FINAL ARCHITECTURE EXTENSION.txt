# Финальное ТЗ на доработку VIREL Back Office до уровня полноценной операционной платформы

## 0. Статус документа

Это финальная рабочая версия ТЗ для разработки.
Документ описывает, что именно нужно реализовать поверх уже существующей системы Virel v3 и как это должно работать в продакшне.

База для этого ТЗ уже существует:

* архитектурная спецификация Back Office AuraEscort с 21 модулем, моделью данных, state machines, API, метриками, нефункциональными требованиями, MVP и Phase 2 
* rebuild plan для Virel v3 с анализом текущего состояния, gap analysis и фазами внедрения 
* детальные phase-документы по Phase 1, Phase 2, Phase 3, Phase 4 и design refresh, где уже зафиксированы направления по Prisma schema, UI, RBAC, audit, pricing, membership, SLA, data quality и другим модулям

Этот документ не отменяет текущую рабочую систему. Он задаёт, как её нужно расширить, не ломая уже работающее ядро.

---

# 1. Цель проекта

## 1.1. Главная цель

Доработать Virel v3 так, чтобы система перестала быть просто административной панелью и стала полноценным back office, который управляет:

* лидами
* клиентами
* бронированиями
* оплатами
* моделями
* доступностью
* потерями выручки
* инцидентами
* рисками
* membership
* удержанием клиентов
* альтернативными предложениями
* уведомлениями
* фоновыми задачами
* аналитикой владельца
* дисциплиной и эффективностью персонала

## 1.2. Что должна уметь система после завершения работ

После реализации система должна позволять владельцу и операционной команде отвечать на вопросы:

* кто этот клиент и какая у него полная история
* сколько денег этот клиент уже принёс
* сколько лидов теряется и почему
* где деньги утекают из-за ресепшена, модели, клиента или системы
* почему не подтвердился конкретный booking
* кто из сотрудников реально приносит выручку, а кто создаёт потери
* какие клиенты нужно срочно удерживать
* какие альтернативы надо предлагать, если модель недоступна
* какие операции требуют срочного вмешательства прямо сейчас
* какие уведомления не ушли
* какие фоновые процессы зависли
* где высокий риск фрода
* где перегруз по спросу и нехватка моделей

## 1.3. Что нельзя допустить

Нельзя строить новую логику поверх хаоса.
Нельзя добавлять сущности без понятных связей.
Нельзя описывать бизнес-логику словами без статусов, полей, правил и API.
Нельзя оставлять места, где разработчик будет трактовать ТЗ по-своему.

# 2. Исходное состояние системы

## 2.1. Что уже есть и должно быть сохранено

По rebuild plan и phase-документам в системе уже есть рабочее ядро: Lead Management, Booking Management, базовый Payment & Finance, Model Control, RBAC, Audit Trail, Integrations Hub, частичный SEO, частичный Onboarding и базовый SLA/Tasks. Также уже зафиксировано, что существующие state machines в Virel местами детальнее базовой спеки и их нельзя упрощать ради формального совпадения.

## 2.2. Что не ломать

Существующие рабочие процессы нельзя переписывать с нуля.

Правило:
все изменения аддитивные.

Это означает:

* новые поля добавляются с default values или nullable
* новые таблицы не ломают существующие связи
* существующие записи в базе остаются валидными после миграции
* state machines расширяются, а не упрощаются
* audit trail сохраняется
* RBAC расширяется, а не выкидывается

# 5. Общие правила реализации

## 5.1. Правило данных

Все денежные значения хранить в Decimal / Numeric, не в Float.

### Почему

Float приводит к ошибкам округления в:

* payments
* refunds
* membership discounts
* lost revenue
* compensation
* pricing

### Денежные поля, которые обязаны быть Decimal / Numeric

* total_spent
* amount
* value_amount
* base_price
* final_price
* refunded_amount
* payout_amount
* compensation_amount
* price_monthly
* discount_amount
* lost_revenue_amount

## 5.2. Правило именования

В БД использовать snake_case.
В Prisma / TypeScript допускается camelCase mapping, но единообразно по проекту.

Пример:

* client_id в БД
* clientId в Prisma

## 5.3. Правило soft delete

Hard delete запрещён для ключевых бизнес-сущностей.

### Обязательный soft delete

* clients
* bookings
* models
* reviews
* incidents
* membership_plans
* pricing_rules
* notification_templates

### Необязательный soft delete

* jobs
* logs
* events
* feed_items

## 5.4. Правило аудита

Любая критичная операция должна создавать AuditLog.

### Критичные операции

* merge clients
* change booking status
* refund payment
* change client risk status
* change model status
* create / resolve incident
* resolve lost revenue entry
* change pricing rules
* create / cancel membership
* manual override availability
* manual close operations feed item

## 5.5. Правило фоновых процессов

Любая операция дольше 500 мс должна быть вынесена в Job Queue.

### Примеры

* notification sending
* report generation
* fraud scan
* onboarding processing
* SEO generation
* retention scan
* pricing recalculation
* media processing

## 5.6. Правило RBAC

Все новые API и все новые экраны обязаны проверять доступ по ролям.

## 5.7. Правило идемпотентности

Все state-changing endpoints, где возможен повторный вызов, должны быть идемпотентными.

### Примеры

* refund
* create membership
* payment webhook
* notification dispatch
* retry job
* merge clients

## 5.8. Правило reason_code

Для ручных действий, влияющих на деньги, статус, риск или доступность, обязателен reason_code.

## 7.2. ClientIdentity

### Назначение

Хранить все идентификаторы клиента и предотвращать дубли.

### Поля

* id
  Уникальный идентификатор записи identity.

* client_id
  Ссылка на Client.

* identity_type
  Тип идентификатора.

* raw_value
  Оригинальное значение, как пришло из источника.

* normalized_value
  Нормализованное значение для поиска совпадений.

* is_primary
  Основной ли это идентификатор для клиента.

* is_verified
  Подтверждён ли этот идентификатор.

* confidence_score
  Насколько система уверена в валидности этой identity.

* source
  Откуда identity пришла.

* verified_at nullable
  Когда identity была подтверждена.

* last_seen_at nullable
  Когда identity в последний раз использовалась.

* is_active
  Активна ли identity после merge.

* created_at
  Дата создания.

* updated_at
  Дата обновления.

### identity_type

* phone
* email
* telegram
* whatsapp
* device
* ip
* card_fingerprint

### source

* manual
* inquiry
* booking
* payment
* integration
* system

### Правила

1. Все новые inquiry, bookings и payments должны проверять identity на совпадение.
2. Совпадение по normalized_value должно либо привязать к существующему клиенту, либо создать merge candidate.
3. После merge duplicate identities не удаляются, а помечаются inactive.
4. Для одной client identity type + normalized_value не должно быть дубликатов среди active identities.

# 26. API ТРЕБОВАНИЯ

## 26.1. Общий формат ответа

Успех:

```json
{ "data": { ... } }
```

Ошибка:

```json
{ "error": { "code": "...", "message": "..." } }
```

Этот формат уже закреплён в спецификации и должен быть сохранён. 

## 26.2. Обязательные требования ко всем новым endpoints

Для каждого нового endpoint обязательно:

1. RBAC check
2. input validation
3. pagination для list endpoints
4. sorting support, где это уместно
5. filters by date range, где это уместно
6. audit logging для critical actions
7. idempotency для state-changing operations
8. standardized error codes
9. permissions-aware responses

# 30. МЕТРИКИ И OWNER ANALYTICS

## 30.1. Сохранить из исходной спеки

В исходной архитектуре уже есть Revenue, Net Margin, Conversion, ARPU, MRR, Churn Rate, Chargeback Rate, SLA Breach Rate и другие. Они должны быть сохранены. 

## 30.2. Добавить новые метрики

### Lost Revenue Amount

Сумма потерь по LostRevenueEntry за период.

### Lost Revenue by Root Cause

Разбивка потерь по root_cause.

### Alternative Offer Acceptance Rate

Доля принятых alternative offers.

### Retention Recovery Rate

Доля retention actions, приведших к новому booking.

### Client Duplicate Rate

Доля клиентов, у которых были merge candidates или merge operations.

### Jobs Failed Rate

Доля jobs, завершившихся failed или dead_letter.

### Notification Delivery Success Rate

Доля успешных уведомлений по Notification и NotificationLog.

### Staff Conversion Ranking

Рейтинг сотрудников по conversion.

### Staff Lost Revenue Ranking

Рейтинг сотрудников по amount lost.

### Availability Conflict Count

Количество конфликтов доступности.

### Membership Benefit Cost

Стоимость скидок и преимуществ membership.

### Risk Escalations per Week

Количество клиентов, переведённых в monitoring / restricted / blocked.

### VIP Client Share of Revenue

Доля выручки от VIP / whale клиентов.

### Win-back Conversion Rate

Доля клиентов, вернувшихся после retention action.

## 30.3. Для каждой метрики обязательно указать

* source table
* source fields
* recompute schedule
* cache / materialization strategy

### Пример

Lost Revenue Amount:

* source: lost_revenue_entries
* fields: amount, status, detected_at
* recompute: hourly
* materialization: daily summary table

# 32. ACCEPTANCE CRITERIA

## 32.1. Identity

1. merge не теряет связанные записи
2. merge создаёт audit log
3. duplicate client не удаляется hard delete
4. duplicate candidates отображаются в UI

## 32.2. Timeline

1. минимум 10 основных событий автоматически пишутся в ClientEvent
2. timeline сортируется строго по created_at desc
3. timeline виден в профиле клиента

## 32.3. Availability

1. система отклоняет overlapping confirmed bookings
2. auto-created cooldown / travel slots создаются
3. manual block работает и логируется
4. conflicts сохраняются в журнале

## 32.4. Lost Revenue

1. минимум 5 автосценариев реально срабатывают
2. entries нельзя удалять
3. owner видит summary и root causes

## 32.5. Operations Feed

1. high / critical items видны на Action Center
2. item можно assign / acknowledge / resolve
3. duplicate feed items не штампуются бесконечно

## 32.6. Notifications

1. уведомления идут через Notification Engine, а не напрямую из модулей
2. failed notifications попадают в retry flow
3. duplicate notifications не создаются при повторном webhook

## 32.7. Jobs

1. dead-letter jobs видны в UI
2. retry работает
3. hanging jobs видны по heartbeat

## 32.8. Retention

1. at-risk scan работает по расписанию
2. retention action может быть связан с booking conversion
3. at-risk clients видны в UI

## 32.9. Alternative Offers

1. при недоступной requested model система генерирует alternatives
2. offer acceptance / rejection логируется
3. offer conversion считается

## 32.10. Staff Performance

1. snapshot не редактируется вручную
2. owner видит raw metrics и final score
3. лидерборд работает по периодам
