## ГДЕ У ВСЕХ ДЫРЫ

Это места где мы их обгоняем:

**Контент написан для робота.** Переспам, шаблонные фразы, нечитаемый текст. Google с 2022 года (Helpful Content Update) явно депрессирует такие страницы. Ни один конкурент не адаптировался.

**Schema слабая или отсутствует.** Dolls & Roses — lang="en-US" на британском сайте. Большинство — никакой Schema кроме базовой. Rich snippets никто не использует.

**Внутренняя перелинковка не выстроена.** Страницы существуют изолированно. Нет логики передачи веса от сильных страниц к слабым.

**Блог не работает как воронка.** Статьи не ведут на коммерческие страницы, не собирают трафик из информационных запросов в коммерческий интент.

**404 на коммерческих запросах.** Dior — /gfe-london-escorts/ отдаёт 404. Rachael's — /gfe-escorts/ отдаёт 404. Это потерянный трафик который можно подобрать.

**Мобильная версия вторична.** Babylongirls, Hot Collection — сделаны под десктоп. 70%+ трафика ниши — мобильный.

**Нет страниц под информационный интент.** Никто не закрывает запросы типа "what is girlfriend experience", "how to book escort london", "escort agency vs independent". Это тысячи переходов в месяц которые никто не берёт.

## ЧТО ОСТАВЛЯЕМ, УСИЛИВАЕМ, УБИРАЕМ

**Оставляем:**
- Текущую структуру /london/ для локаций
- /services/ раздел (62 страницы — хорошая база)
- Next.js + Vercel (техническое преимущество по Core Web Vitals)
- Find Your Match (уникальная механика, нигде нет)
- Schema.org которая уже есть (расширить)

**Усиливаем:**
- Контент на каждой локационной странице: с 216 слов до 1500+
- Добавляем категорийный раздел /categories/ с нуля
- Блог: с 6 до 60+ статей за 3 месяца
- Внутреннюю перелинковку между локациями, категориями и сервисами
- Schema на каждом типе страниц (LocalBusiness, BreadcrumbList, FAQPage, Article)

**Убираем:**
- Дубли /london/X и /escorts-in/X — оставляем одну структуру, остальное 301 редирект
- lang="en" меняем на lang="en-GB"
- canonical который сейчас указывает на vercel.app — поменять при переезде

**Не делаем:**
- Страниц по станциям метро (риск doorway)
- Категорию "cheap escorts" (убивает позиционирование)
- Покупных ссылок на старте (риск в нише)

## ЭТАП 1 — ДО РЕГИСТРАЦИИ ДОМЕНА

Всё что здесь — делается на vercel.app. SEO-вес не накапливается, но архитектура и контент готовятся к переезду.

**Архитектура URL (финальная):**

```
/                          — главная
/companions/               — каталог
/companions/[slug]/        — профиль

/london/[district]-escorts/          — локации (45+ страниц)
/categories/[category]-escorts/      — категории (30+ страниц) ← сейчас 0
/services/[service]/                 — сервисы (62 страницы)
/blog/[slug]/                        — блог

/how-it-works/
/about/
/contact/
/faq/
/privacy/
/terms/
```

**Категории — приоритетный список (создать сейчас):**

russian-escorts, blonde-escorts, brunette-escorts, busty-escorts, slim-escorts, mature-escorts, young-escorts, asian-escorts, european-escorts, gfe-escorts

Tier B (средний спрос):

french-escorts, italian-escorts, spanish-escorts, brazilian-escorts, redhead-escorts, tall-escorts, model-escorts, student-escorts, tantric-escorts, travel-companion

**Шаблон Title/H1/Meta для каждого типа страниц:**

Локация: `Escorts in [District] London | Virel Companion Agency`
H1: `[District] Escorts London`
Meta: `Premium companions available in [District], London. Verified, discreet. Book with Virel.`

Категория: `[Category] Escorts London | Virel`

Сервис: `[Service] London | Virel Companion Agency`
H1: `[Service] in London`

H1: `[Topic]`

**Минимальный объём контента по типам:**

Локация: 1200-1500 слов. Структура: что за район, лучшие отели, рестораны, доступные компаньоны, FAQ (5 вопросов), внутренние ссылки на соседние районы и категории.

Категория: 800-1000 слов. Структура: описание типа, что клиент получает, примеры компаньонов, FAQ, ссылки на смежные категории и локации.

Сервис: 600-800 слов. Структура: что это, как работает, с кем доступно, FAQ.

Блог: 1000-1500 слов. Структура: реальный ответ на поисковый вопрос, 1-2 ссылки на коммерческие страницы.

**Schema по типам:**

Главная: Organization + LocalBusiness + WebSite

Локация: LocalBusiness + BreadcrumbList + FAQPage

Категория: CollectionPage + BreadcrumbList + FAQPage

Профиль: Person + BreadcrumbList

Блог: Article + BreadcrumbList + FAQPage

**Система внутренних ссылок:**

Каждая локационная страница ссылается на: 3-4 соседних района, 5-6 популярных категорий, 3-4 сервиса.

Каждая категорийная страница ссылается на: топ-5 локаций, смежные категории, 3-4 профиля компаньонов.

Главная ссылается на: топ-10 локаций, топ-8 категорий, топ-6 сервисов, последние 3 статьи блога.

Блог ссылается на: 1-2 коммерческие страницы релевантные теме статьи.

**Приоритет контента для создания до домена:**

1. 10 категорий Tier A (по шаблону, 800-1000 слов каждая)
2. 15 топовых локаций (Mayfair, Knightsbridge, Chelsea, Kensington, Belgravia + 10 других) по 1200-1500 слов
3. How it works, About, FAQ — коммерческие страницы доверия
4. 10 статей блога — информационный трафик + воронка на коммерческие страницы

## ЭТАП 2 — СРАЗУ ПОСЛЕ ЗАПУСКА ДОМЕНА

День 1:
- Регистрация в Google Search Console на новый домен
- Отправка sitemap.xml (разбитый по типам: sitemap-locations.xml, sitemap-categories.xml, sitemap-services.xml, sitemap-blog.xml, sitemap-profiles.xml)
- Проверка robots.txt — убедиться что /companions/[slug]/ закрыт от индексации (профили компаньонов не нужны в индексе по юридическим соображениям, только если нет явной необходимости)
- Проверка всех canonical — указывают на финальный домен, не на vercel.app
- 301 редиректы с vercel.app на новый домен

Неделя 1:
- Mobile-Friendly Test через Google Search Console
- Core Web Vitals — проверить LCP, CLS, INP (Next.js даёт хорошую базу, но нужно проверить конкретно)
- Rich Results Test — проверить Schema на главной, одной локации, одной категории
- Проверка дублей через Coverage report в Search Console
- lang="en-GB" — убедиться что стоит во всех шаблонах

## ЭТАП 3 — РОСТ (1-12 МЕСЯЦЕВ)

**Быстрые победы (первые 2-4 недели после запуска):**

Поднять контент на топ-10 локациях с текущих 216 слов до 1200+. Это самое быстрое движение по локальным запросам — страницы уже проиндексированы, нужно только углубить контент.

Создать 10 категорий Tier A. Сейчас их нет совсем — каждая новая страница это новый поисковый запрос который был полностью упущен.

Исправить дубли URL. /london/X vs /escorts-in/X — 301 редирект, один канонический путь.

**Рост через 1-3 месяца:**

30 категорий Tier A+B — полный охват категорийного спроса

45 локационных страниц с полным контентом

20 статей блога ориентированных на информационные запросы с воронкой на коммерческие страницы

Страницы /how-it-works/, /about/, /discretion/ — доверие + информационный трафик

FAQPage Schema на всех коммерческих страницах — шанс на FAQ-сниппеты в результатах поиска

**Фундамент на 6-12 месяцев:**

60+ статей блога — авторитет по теме, органические ссылки

London Guide раздел — гиды по районам, отелям, ресторанам. Трафик из смежных запросов + потенциальные внешние ссылки от lifestyle-сайтов

Страницы по национальностям (/russian-companions/, /italian-companions/) — Dolls & Roses показали что это работает (2289 слов на /russian-escorts/)

Расширение профилей компаньонов — языки, образование, специализация. Это влияет на время на сайте и косвенно на ранжирование

Регулярность блога — минимум 4 статьи в месяц. Google видит активный сайт.

## РЕАЛИСТИЧНАЯ ХРОНОЛОГИЯ РЕЗУЛЬТАТОВ

Месяц 1-2 после запуска домена: индексация, первые позиции по низкочастотным запросам (escorts in [specific district], [nationality] escorts london).

Месяц 3-4: топ-10 по среднечастотным категорийным запросам (russian escorts london, blonde escorts london).

Месяц 5-6: конкуренция по топовым локационным запросам (escorts in Mayfair, Knightsbridge escorts).

Топ-1 по "london escorts" за 2 месяца — нереально. За 9-12 месяцев при правильном выполнении плана — реально.

## ОДНА ГЛАВНАЯ МЫСЛЬ

Конкуренты слабы не потому что плохие — они слабые потому что строились в 2010-2015 годах и не перестраивались. Google изменился. Helpful Content, E-E-A-T, Page Experience — всё это работает против старых сайтов с тонким контентом и шаблонными текстами.

Virel строится сейчас, с нуля, на правильном стеке, с правильной архитектурой. Это само по себе преимущество если не повторять их ошибки.
