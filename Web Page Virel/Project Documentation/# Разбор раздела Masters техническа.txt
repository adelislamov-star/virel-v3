# Разбор раздела Masters: техническая архитектура и применение к профилям

Итак, я прошёлся по всем семи разделам и вижу всю картину целиком. Давайте разберём каждый по косточкам — как это сделано технически, как связано с профилями, и что стоит брать как образец, а что выбросить.

---

## Архитектура: как Masters связан с профилями

Всё работает на классической реляционной модели: Masters — это справочники/словари. Профиль (Staff) подтягивает значения из этих словарей через внешние ключи (FK) и связи many-to-many. Вот полная цепочка:

```
County (1) → Locations (many) → Staff.main_location (FK)
Category Groups (1) → Categories (many) → Staff (many-to-many через pivot)
Services (many) → Staff (many-to-many через pivot)
Attributes (1) → Attribute Values (many) → Staff.stats (FK на каждый атрибут)
Call Rate Masters (1) → Staff Rates (many, переопределяются на уровне профиля)
```

Это стандартный паттерн Laravel Eloquent с промежуточными таблицами. Реализован правильно.

## Разбор по разделам

### 1. Category Groups (6 групп)
**Технически**: простейший справочник с полями Name, Description, Sort Order, Status. Никакой связи с фронтендом кроме логической группировки Categories. Timestamps присутствуют.

**Как используется в профиле**: напрямую не используется — только косвенно через Categories. Группировка нужна для фильтров на фронте (например, показывать сервисы в разделах «Kinky», «Service», «Fetish»).

**Что взять**: идея двухуровневой таксономии Services (Groups → Categories) — правильная архитектура. Позволяет создавать блоки на странице профиля без хардкода.

**Что не брать**: поле Categories в списке показывает **0 для всех групп** — это значит, что Categories не привязаны к Category Groups через UI. Связь либо сломана, либо не реализована в интерфейсе. Это архитектурная дыра — данные в базе могут быть, но интерфейс их не отображает.

---

### 2. Services (64 услуги)
**Технически**: полноценная сущность с Title, Slug, Image, Introduction Text (rich text), Description (rich text), Status (active/popular). Плюс SEO-вкладка с заполненными Title, Description, Keywords. Это самый проработанный раздел в Masters.

**Как используется в профиле**: в разделе GROUP профиля (табе «Group») escort может быть привязан к Services через чекбоксы.

**Что взять**:
- Двойной текст: Introduction (короткий) + Description (длинный rich text) — правильная структура для SEO-страниц услуг
- SEO-вкладка **реально заполнена** (в отличие от профилей): title "69 Escorts London | Mutual Oral Pleasure | Muse Escorts" — конкретный, с ключевыми словами и ценой
- Slug генерируется автоматически, но редактируется — хороший паттерн
- Image per service — правильно, визуальная идентификация на фронте
- Флаг Popular — позволяет выводить топ-сервисы в отдельном блоке

**Что не брать**:
- **Изображение для сервиса "69" — фотография парковки с автомобилями** (carrental-services.jpg, 367 KB). Это дефолтная заглушка от платформы, загруженная случайно и никем не замеченная. Из 64 сервисов, вероятно, большинство с такими же заглушками — это нужно проверить массово.
- Флаг называется "Popular Location" для сервиса — терминологическая ошибка в коде. Должно быть "Popular Service". Это копипаст из Location модели.

### 3. Categories (14 категорий)
**Технически**: самый богатый раздел. Title, Slug, **отдельное H1 heading** (независимое от Title!), Introduction Text (rich text), Description (rich text с подзаголовками), Status, Popular. SEO-вкладка. Двухпанельный layout (контент слева, статус и timestamps справа).

**Как используется в профиле**: в GROUP-табе профиля — привязка к категориям (Selfie, Video, Busty, Mature и т.д.).

**Что взять** — это лучший пример реализации во всём Masters:
- **Разделение Title и H1 heading** — правильно и важно! Title используется в `<title>` тега, H1 — в заголовке страницы. Это позволяет, например, иметь Title "Selfie" и H1 "Selfie Escorts London". Именно такой подход должен быть везде, включая профили.
- Slug "selfie-escorts" (не просто "selfie") — slug уже содержит ключевое слово. Правильный SEO-паттерн.
- Introduction + Description — два уровня контента. Introduction идёт в первый абзац страницы (важно для ботов), Description — основной SEO-текст.
- SEO-вкладка с реальным наполнением.
- Двухколоночный layout формы — эффективное использование экрана.

**Что не брать**:
- Нет связи с Category Groups в интерфейсе (видели в списке — счётчик 0). Нужно добавить поле «Category Group» в форму Category.
- Нет поля для OG Image (соцсети).

### 4. Attributes (15 атрибутов)
**Технически**: самая нестандартная и сложная сущность. Attribute — это EAV-паттерн (Entity-Attribute-Value). Каждый атрибут (Age, Height, Weight, Dress Size, Bust Size, Bust Type, Hair Colour, Eye Colour, Smoking Status, Tattoo Status...) имеет свой набор допустимых значений. Каждое значение — это строка + опциональный internal link slug. Флаг "Is Multiple" определяет, можно ли выбрать несколько значений.

**Как используется в профиле**: вкладка Stats в профиле — это динамически генерируемая форма на основе активных Attributes. Добавил новый атрибут в Masters → он автоматически появится на всех профилях.

**Что взять**:
- Идея **EAV для физических параметров** — правильная архитектура для гибкости. Не нужно менять схему БД при добавлении нового поля профиля.
- Поле "link" при каждом значении атрибута (например, возраст 18 → ссылка `/18-year-old-escorts`) — это элегантное решение для SEO-фильтрации. Каждое значение становится потенциальной landing page.
- Is Multiple флаг — позволяет мультивыбор для некоторых атрибутов (например, National можно быть British + Latina).
- Drag & drop сортировка значений внутри атрибута.

**Что не брать / переделать**:
- **Age как атрибут с 33 значениями (18, 19, 20... 50)** — это антипаттерн. Возраст должен быть числовым полем с валидацией min/max, а не справочником с 33 записями. Текущий подход: добавить возраст 51? Нужно идти в CMS и добавлять строку вручную. Это не масштабируется.
- **Height с 41 значением** и **Weight с 51 значением** — та же проблема. Это должны быть числовые слайдеры или поля с единицами измерения (см/кг), а не фиксированные списки.
- Нет группировки атрибутов. 15 атрибутов в одном списке без секций — при 30+ атрибутах это станет неуправляемым.
- Нет возможности задать тип данных атрибута (text/number/boolean/date). Всё хранится как строка.

### 5. Call Rates (8 позиций)
**Технически**: справочник тарифных позиций. Label, Default Incall Rate, Default Outcall Rate, Currency (GBP hardcoded), Display Order, Active. Это шаблон — конкретные цены устанавливаются на уровне каждого профиля (Staff → Rates tab), но базируются на этом справочнике.

**Как используется в профиле**: вкладка Rates — там отображаются именно эти 8 позиций с возможностью переопределить цену и добавить/убрать позицию. Функция "Copy Rates from" — скопировать тариф с другого профиля.

**Что взять**:
- Двухуровневая система: глобальные Master rates → индивидуальные Staff rates — правильный паттерн наследования с возможностью override.
- Copy Rates from — отличная функция для операционной эффективности. Добавить 20 новых профилей? Выбрал шаблонный профиль → скопировал тарифы.
- Display Order — управление порядком отображения позиций.

**Что не брать**:
- **Default Incall Rate и Default Outcall Rate пустые** у всех 8 позиций в Masters. Смысл в том, что каждый профиль устанавливает свои цены, но тогда поля Default Rate в Masters бессмысленны — они нигде не используются. Это мёртвый функционал, создающий путаницу.
- Currency hardcoded как "GBP" — нет возможности сменить на USD, EUR. Для мультирегионального проекта это ограничение.
- Нет поля "Minimum duration" или логики ценообразования (например, обязательная минимальная продолжительность).

---

### 6. Counties (1 запись: Greater London)
**Технически**: верхний уровень геоиерархии. County → Locations. Title, ISO Code, Status, Popular, счётчик Locations (31). Timestamps с датой "Dec 19, 2025".

**Как используется в профиле**: косвенно — Location привязана к County, а профиль привязан к Location.

**Что взять**:
- Правильная геоиерархия: County → Location → Staff. Масштабируется на Манчестер, Бирмингем, Эдинбург — просто добавляешь новый County и создаёшь Locations под него. Всё остальное подтянется.
- ISO Code поле — хорошая практика для машинного распознавания региона.

**Что не брать**:
- Одна запись с 2025 года — вся иерархия County не используется по назначению. Если сайт планирует выйти за пределы Лондона, нужно было с самого начала строить контент именно через County, а не один глобальный "Greater London".

---

### 7. Locations (31 запись)
**Технически**: самый богатый справочник из Masters. County (FK + возможность создать inline), Title, Slug (с -escorts суффиксом!), H1 Heading, Introduction Text, Description (rich text), Status, Popular Location. SEO-вкладка с заполненным Title, Description (keywords пусты). Двухколоночный layout.

**Как используется в профиле**: поле "Main Location" в Profile-табе профиля — dropdown привязки к конкретному Location. Отображается на карточке профиля как "Chelsea/ £300.00".

**Что взять**:
- **Slug с суффиксом -escorts** (baker-street-escorts) прямо в поле — это SEO-решение на уровне архитектуры, не URL-конфигурации. Правильно.
- **H1 отдельно от Title** — как и в Categories, это best practice. "Baker Street" vs "Baker Street Escorts".
- Introduction Text с подсказкой "Brief introduction text that appears at the top of the location page" — хорошая UX-документация прямо в форме.
- SEO Title заполнен правильно: "Baker Street Escorts W1 | Incalls & Outcalls | Muse Escorts" — содержит постальный код W1, что важно для локального SEO.
- Inline создание County через кнопку "+" рядом с dropdown — удобно, не нужно выходить из формы.

**Что не брать**:
- Popular Location флаг включён только у 0 из 31 локаций (все красные в списке). Либо функционал не используется, либо все забыли его включить. Нужно аудит.
- SEO Keywords пустые у всех Locations — та же проблема, что у профилей (хотя Keywords и не важны для Google, пустота сигнализирует о небрежности).
- Нет поля для географических координат (lat/lng) — нет возможности показывать карту или Local Business schema для SEO.

## Итоговая оценка: что брать, что не брать

**Брать как образец:**

Двухуровневую таксономию Services (Groups → Services) для гибкой фильтрации профилей. Отдельное поле H1 heading независимо от Title — применить везде, включая в профилях. Slug с семантическим суффиксом прямо в Master-записи (baker-street-escorts, selfie-escorts). EAV-подход для физических параметров — гибко, но только для нечисловых параметров (Hair Color, Eye Color, Nationality). Двухуровневое ценообразование Master → Override на уровне профиля. Функцию Copy Rates across profiles. Inline-создание связанных сущностей через кнопку "+" в dropdown. Introduction + Description как два уровня контента на каждой странице.

**Не брать / переделать:**

EAV для числовых параметров (Age 18–50, Height, Weight) — заменить числовыми полями с range-валидацией. Пустые Default Rates в Masters — либо использовать как реальный шаблон по умолчанию, либо убрать поля. Связь Category Groups → Categories не отображается в интерфейсе — починить или убрать уровень Group. County как надстройка пока не используется — если нет планов выхода за пределы Лондона, это лишний слой навигации. Отсутствие OG Image и geo-координат в Locations — добавить для полноценного Local SEO. Флаги Popular Location / Popular Category — нужно либо использовать всеми записями, либо убрать как мёртвый функционал.
