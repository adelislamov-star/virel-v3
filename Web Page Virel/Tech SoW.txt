## 4. Информационная архитектура и URL стратегия

Это раздел, который предотвращает главную SEO катастрофу: бесконечные комбинации фильтров, дубли и мусор в индексе.

### 4.1 Базовые правила URL

* Публичные URL только человекочитаемые.
* UUID запрещены в публичных URL (у Aura это критическая ошибка). 
* Изменение slug после публикации запрещено по умолчанию. Если менять, то только через админку с автоматическим 301.

### 4.2 Структура URL

Обязательная структура (финальная подтверждается на IA этапе, но логика фиксирована):

1. Главный листинг:

* /london-escorts

2. Пагинация:

* /london-escorts/page/2
* /london-escorts/page/3

3. Geo страницы:

* /escorts-in-mayfair
* /escorts-in-kensington
  и т.д.

Список приоритетных районов из твоего ТЗ на запуск:
Mayfair, Kensington, Knightsbridge, Chelsea, Belgravia, Marylebone, Westminster, Notting Hill, Soho, Covent Garden, Canary Wharf, City of London. 

4. Services:

* /services/{service-slug}

5. Attributes:

* /{attribute}-escorts-london
  пример:
* /blonde-escorts-london

6. Профиль модели:

* /catalog/{slug} (как в твоем ТЗ) 

### 4.3 Политика индексируемости (строго)

#### 4.3.1 Indexable (в sitemap)

* Главная
* /london-escorts и ее пагинация
* Geo страницы из whitelist
* Services страницы из whitelist
* Attributes страницы из whitelist
* Профили моделей в статусе Published
* Блоговые статьи (если включены)
* FAQ, About, Contact, Legal

#### 4.3.2 Non-indexable (не в sitemap)

* любые произвольные комбинации фильтров, не входящие в whitelist
* админка и служебные маршруты
* поиск, избранное, любые персональные страницы
* query параметрные страницы, если они не утверждены в whitelist

Механика:

* Non-index страницы получают meta robots: noindex,follow
* canonical указывает на базовую indexable страницу (обычно /london-escorts или соответствующий whitelist листинг)
* такие страницы не попадают в sitemap.

Важно: у Aura некорректная SEO пагинация (infinite scroll без ссылок) и это убивает crawl. У нас так быть не может. 

### 4.4 Пагинация

Требования:

* пагинация обязана иметь HTML ссылки <a href> на страницы.
* страницы пагинации должны быть доступны без JS.
* стратегия canonical:

  * canonical каждой page/N указывает на саму себя
  * index,follow для page/N
  * запрет canonical page/N на page/1, чтобы не потерять обнаружение карточек через глубину каталога.

Причина: если canonical всех page/N на page/1 и при этом карточки “висят” глубоко, ты теряешь часть профилей по crawl discoverability.

### 4.5 Редиректы и миграция

По твоему ТЗ:

* нужна mapping table UUID -> slug и 301 редиректы /catalog/{UUID} -> /catalog/{slug}. 
  Требование:
* таблица редиректов хранится в БД
* выгружается в файл на сервере и/или применяется на уровне edge (Cloudflare rules) по согласованию.

## 6. Интеграции и протоколы

Этот раздел специально “закручиваю”, чтобы не было дублей и потерянных статусов.

### 6.1 Telegram боты (как в твоем ТЗ)

Компоненты:

* DivaReceptionBot: уведомления reception, 30 минутные напоминания, эскалация Tommy при отсутствии ответа. 
* Kesha ZeroGap Bot: синхронизация бронирований с AppSheet, карточки, парсинг datetime/duration, read-only внешние данные и локальное хранение в SQLite. 

Требования:

* Все уведомления идут через очередь (Redis) с retry.
* Идемпотентность: на одно событие 1 сообщение в 1 канал. Повторный вызов не создает дубль.
* Логирование доставки: sent/failed, reason, attempts.

### 6.2 AppSheet API

В твоем ТЗ указано:

* двусторонняя синхронизация бронирований
* webhook для обновления статусов в реальном времени
* авто обновление календаря занятости. 

Фиксируем источник истины:

* Источник истины по бронированиям: AppSheet (если это текущая операционная база).
* Сайт создает “заявку” и передает в AppSheet как запись.
* AppSheet меняет статусы, сайт подтягивает изменения.

Протокол синхронизации:

1. Create:

* сайт отправляет booking_create в AppSheet
* получает external_id (appsheet_booking_id)
* сохраняет mapping.

2. Update:

* AppSheet вызывает webhook на сайт при изменении статуса или календаря.
* если webhook недоступен, сайт раз в X минут (настраиваемо) делает reconciliation pull.

3. Idempotency:

* все запросы имеют request_id
* повторный request_id не создает новую запись.

4. Failover:

* если AppSheet недоступен:

  * сайт сохраняет заявку локально со статусом “queued_for_sync”
  * уведомляет reception
  * при восстановлении делает sync.

### 6.3 CRM (опционально)

* если CRM включаем: webhook события booking_created и booking_confirmed
* дедупликация по телефону/мессенджеру
* экспорт CSV/Excel из админки обязателен.

## 8. SEO и техническая оптимизация

### 8.1 Canonical

* canonical на 100% страниц обязателен. 
* для whitelist страниц canonical на себя
* для non-index фильтров canonical на базовую страницу

У Aura canonical отсутствует на 100% страниц, это причина дублей и проблем индексации. 

### 8.2 Sitemap

* автогенерация
* sitemap index + разбиение по типам
* включает все indexable URL и только их
* обновляется при публикации/архивировании профиля

У Aura sitemap содержит 5 URL вместо 144 и это прямой провал индексации. 

### 8.3 Robots.txt

* корректный домен .co.uk
* запрет админки и служебных
* ссылка на sitemap index

У Aura были доменные ошибки Host/sitemap. Это запрещено. 

### 8.4 Structured Data (Schema) обязательна

У Aura Schema отсутствует на всех типах страниц.

## 10. Производительность, CWV и фронтенд правила

Цели CWV закреплены в твоем ТЗ:
LCP < 2.5s, INP < 200ms, CLS < 0.1. 

Что именно делаем, чтобы не повторить Aura:

* медиа:

  * 100% WebP и AVIF (AVIF обязателен, у Aura 0%) 
  * lazy loading для всех изображений ниже первого экрана (у Aura 0%) 
  * фиксированные размеры для предотвращения CLS
* JS:

  * минимизация бандла, code splitting
  * запрет тяжелых трекеров на первых миллисекундах
* SSR/ISR:

  * money pages и профили только SSR/ISR, без “пустой страницы пока JS загрузится”
* веса страниц:

  * никакого 7–14MB, как у Aura. Цель initial < 2–3MB. 

Mobile UX:

* запрещено user-scalable=no и maximum-scale=1 (у Aura это было проблемой) 
* sticky CTA на mobile обязателен (у Aura отсутствовал)

## 11. Безопасность и конфиденциальность

Требования из твоего ТЗ:

* SSL, HSTS
* AES-256 для PII
* rate limiting
* GDPR compliance
* Cloudflare DDoS/bot
* логирование действий админов. 

Фиксируем детально, чтобы не было “на словах”:

### 11.1 Политика PII

Храним:

* telegram/whatsapp/email/телефон (если введены)
* даты, длительность, выбранная модель
* internal notes

Не храним в логах:

* сырые телефоны
* сырые usernames
* тексты заявок целиком в системных логах
* IP в чистом виде (только hash или truncated)

Retention:

* PII в booking_requests хранится N дней (настраиваемо, по умолчанию 180)
* после срока: обезличивание, оставляем только агрегаты.

### 11.2 Админка

* отдельный домен или маршрут /admin закрыт от индексации
* 2FA желательно
* rate limit login
* audit log на изменения

### 11.3 WAF

* Cloudflare Bot Management минимум
* правила против массового скачивания медиа
* ограничения на API.

## 14. Тест-кейсы приемки (обязательные)

### 14.1 SEO и индексируемость

1. Опубликован профиль модели:

* URL доступен
* canonical есть
* в sitemap появился
* schema валидна

2. Архивирован профиль:

* удален из sitemap
* URL отдает 410 или 404 по выбранной стратегии (фиксируется в Discovery)
* внутренние ссылки на него не показываются в листингах

3. Любая non-whitelist комбинация фильтров:

* noindex,follow
* canonical на базовый листинг
* не попадает в sitemap

4. Пагинация:

* есть <a href> на page/2 и обратно
* page/2 индексируется
* canonical на себя

### 14.2 Performance

* PSI mobile >= 90 на главной, каталоге, типовом профиле, geo странице
* LCP/CLS/INP в цели. 

### 14.3 Booking end-to-end

* create booking на сайте
* запись в AppSheet создана или очередь на sync
* уведомление reception отправлено
* отсутствие дубля при повторной отправке с тем же request_id
* webhook update меняет статус в админке

### 14.4 Безопасность

* rate limit на форму
* защита от простого бот-спама
* /admin закрыт от индексации
* audit log пишет изменения SEO полей
